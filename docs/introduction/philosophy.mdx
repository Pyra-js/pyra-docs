# Philosophy

Pyra.js is built on three core principles that guide every design decision: **Ignite → Create → Flow**.

## 🔥 Ignite

**Start instantly. No waiting.**

Modern development tools should respect your time. Pyra starts your dev server in milliseconds, not minutes. No bundling, no preprocessing — just instant feedback.

```typescript
// Traditional bundler
$ npm run dev
⏳ Bundling... (45s)
⏳ Optimizing... (12s)
✓ Ready in 57s

// Pyra
$ pyra dev
✓ Ready in 52ms
```

The "ignite" phase is about removing friction. When you run `pyra dev`, you should see your app immediately. When you save a file, changes should appear instantly.

<Callout type="tip">
Pyra achieves instant starts by serving source files directly as ES modules during development. No bundling means no wait time.
</Callout>

## ⚙️ Create

**Build without barriers.**

Pyra gets out of your way. Sensible defaults, zero-config TypeScript, automatic path resolution — everything just works.

### Convention Over Configuration

```typescript
// No config needed for most projects
// Just create src/main.ts and go

import { render } from './app';
render();
```

### TypeScript First

Pyra treats TypeScript as a first-class citizen. Your `tsconfig.json` is respected automatically. Path aliases work without extra configuration.

```json
// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

```typescript
// Works immediately in Pyra
import { Button } from '@/components/Button';
```

### Framework Agnostic

Pyra doesn't force you into a specific framework. Use React, Vue, Svelte, or vanilla TypeScript. Pyra adapts to your choice.

## 🌊 Flow

**Stay in the zone.**

Great tools disappear. Pyra's HMR is so fast and reliable that you forget it's there. Changes appear instantly, state is preserved, and you stay focused on building.

### Precise HMR

Pyra's HMR only updates what changed. If you edit a component, only that component reloads. Your app state stays intact.

```typescript
// Edit this component
export function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>
    Count: {count} {/* Change this text */}
  </button>;
}

// HMR updates the text instantly
// count state is preserved
// No full page reload
```

### Predictable Builds

Development and production should behave the same. Pyra uses the same module resolution in both modes, so you won't encounter "works in dev, breaks in prod" surprises.

<Callout type="note">
Pyra uses native ES modules in dev and Rollup for production. This hybrid approach gives you speed during development and optimization for production.
</Callout>

## Design Principles

### 1. Performance is a Feature

Speed isn't optional. Every millisecond matters. Pyra is obsessed with performance at every layer.

### 2. Simplicity Scales

Simple tools scale better than complex ones. Pyra's API surface is small by design. Less to learn, less to break.

### 3. Defaults Matter

Most projects need the same things. Pyra's defaults cover 90% of use cases. Configuration is for the other 10%.

### 4. Extensibility Without Complexity

Plugins should be powerful but not required. Pyra works great out of the box. Plugins are there when you need them.

### 5. Developer Experience First

Tools should feel good to use. Fast feedback, clear errors, helpful warnings — these aren't nice-to-haves, they're requirements.

## What This Means for You

When you use Pyra, you're choosing:

- **Speed** — Instant starts, instant updates
- **Simplicity** — Minimal config, maximum productivity
- **Reliability** — Predictable behavior, clear errors
- **Flexibility** — Extend when needed, stay simple when not

<Callout type="warn">
Pyra is opinionated. If you need deep customization of every build step, Webpack might be a better fit. Pyra optimizes for the 90% use case.
</Callout>
