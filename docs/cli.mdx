# 🔻 CLI 

Pyra’s command-line interface is the front door to the entire toolchain. It’s the fastest way to scaffold apps, run a dev server with instant HMR, produce optimized builds, and preview production output locally. This page explains each command, the global flags that apply everywhere, and practical examples for day-to-day development and CI pipelines. Understanding how Pyra.js resolves configuration from multiple sources. 

## Getting the CLI

You can invoke the CLI via `npx` without installing anything globally, or you can install `pyra` globally if you plan to use it across many projects. In team settings, adding Pyra as a dev dependency and using `package.json` scripts keeps everyone on the same version.

```bash
# one-off usage
npx pyra@latest dev

# global usage
npm i -g pyra
pyra build

# project-local usage
npm i -D pyra
npm run dev
```

When you run a command in a directory, Pyra treats that directory as the project root unless you pass `--root`. It then searches for a config file (such as `pyra.config.ts`) and merges it with defaults and CLI flags. If a config file isn’t found, Pyra proceeds with sensible defaults so you can still develop and build without boilerplate.

## Global flags 

Every command accepts a common set of flags that control core behavior. CLI flags override your config file, which in turn overrides Pyra’s defaults.

```bash
--root <path>        # set the project root (defaults to cwd)
--config <path>      # point to a specific config file
--mode <name>        # select a mode: dev | build | preview | custom
--logLevel <level>   # silent | error | warn | info | debug
--envPrefix <pref>   # runtime-exposed env var prefixes (comma-separated)
--no-color           # disable ANSI colors in output
--force              # bypass caches where applicable
--help               # show help for the current command
--version            # print the CLI version
```

The `--mode` flag is especially useful in CI or scripted builds where you want to toggle certain options at build time (for example, enabling source maps or switching targets).

## Pyra init

The `init` command gives you a template-based set up. To get started, run:

```bash
pyra init
```

This command will guide you through an interactive setup process:

* **Project name** | specify what to call your project

* **Template** | choose between Vanilla or React (more coming soon)

* **Language** | select TypeScript or JavaScript

Once complete, Pyra will generate a fully configured project with all the right dependencies, boilerplate, and starter code, ready for development or production.

You can also skip the prompts by using flags:

```bash
--template   # Choose the framework (e.g., react or vanilla)
--language   # Choose the language (ts or js)
--pm         # Choose the package manager (npm, pnpm, yarn)
```

Example: 

```bash
pyra init my-app --template react --language ts
```

## Pyra create

The `create` command can quickly scaffold a new Pyra project in seconds. 

```bash
pyra create
# or
pyra create my-app
```
This command will:

* Prompt only for a project name (if not provided)
* Generates a minimal project structure programmatically
* Automatically detects your package manager (or specify manually with --pm)
* Installs dependencies automatically (use --skip-install to skip)

This command is perfect for quick prototyping, minimal setups, or when you just want to get started fast.

Pyra will create a lightweight project with a simple package.json` containing Pyra dependencies, a clean and minimal file structure, and zero framework boilerplate

Use `pyra create` when you want full control, start small, then build your project your way.

Use `--template` to choose between app and library starters, or pass a custom starter name if you’ve published one.

```bash
pyra create my-lib --template library --no-install
```

## Pyra dev

The `dev` command launches the development server with hot module replacement, fast transforms, and clear diagnostics. It watches the file system for changes and updates your browser almost instantly.

```bash
pyra dev
pyra dev --port 5173 --host --open
pyra dev --root ./examples/basic
```

Useful flags include `--host` for LAN testing, `--open` to launch your browser, `--https` for local TLS (honoring server.https in config), and `--port` to avoid conflicts.

## Pyra build 

The `build` command produces a production-ready bundle. It respects `pyra.config.ts` but allows last-mile overrides through flags, which is convenient for CI and one-off experiments.

```bash
pyra build
pyra build --outDir build --minify=false --sourcemap
pyra build --mode staging --target es2020
```

Common flags include `--outDir`, `--target`, `--minify`, `--sourcemap`, and `--clean` (to clear the output directory first). For library authors, a single build can emit multiple formats when configured.

## Pyra preview 

The `preview` command serves the already built output so you can verify production behavior locally. It does not rebuild; it simply hosts your outDir with correct headers and a minimal, reliable server.

```bash
pyra preview
pyra preview --port 5050 --strict
```
Use this before deploying to catch routing or asset path issues that don’t appear in the dev server.

## pyra inspect

The `inspect` command reveals how Pyra sees your project: resolved config, active plugins, final alias table, and the dependency graph. It’s invaluable for debugging configuration mistakes and plugin order.

```bash
pyra inspect
pyra inspect --json > .pyra/inspect.json
```

The `--json` flag prints machine-readable output you can store with your build artifacts.

## Pyra clean

Cleaning removes generated build artifacts and caches tied to the current project root. It’s helpful when switching branches or after large dependency changes.

```bash
pyra clean
pyra clean --all   # remove caches as well as outDir
```

## Practical Examples 

You can combine flags to adapt quickly to different environments.

```bash
# run dev on a custom port and LAN host
pyra dev --port 5175 --host --open

# build without minification for easier debugging
pyra build --minify=false --sourcemap

# preview a specific output directory
pyra preview --root . --port 4173
```

For monorepos, pass `--root` to target a sub-package, or invoke Pyra from workspace scripts with your package manager’s filtering tools.


## Troubleshooting the Cli 

If a command isn’t found, confirm that Node is on your PATH or invoke via `npx`. If the dev server doesn’t reflect changes, clear caches with `pyra clean --all` and restart. When imports fail, verify that your aliases match both your bundler config and any tsconfig path mappings. If preview behaves differently from dev, ensure your `base` path and asset emission settings are correct for production hosting.
