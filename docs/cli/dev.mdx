# pyra dev

Start the Pyra development server for local development with instant Hot Module Replacement.

## Usage

<Terminal>
pyra dev [root]
</Terminal>

## Options

### `--host [host]`

Specify the hostname to listen on. Use `--host` to expose the server to your network.

<Terminal>
pyra dev --host
</Terminal>

```
  PYRA v1.0.0-beta.1  ready in 48 ms

  ➜  Local:   http://localhost:5173/
  ➜  Network: http://192.168.1.100:5173/
```

<Callout type="tip">
Use `--host 0.0.0.0` to listen on all network interfaces, making your dev server accessible from other devices on your network.
</Callout>

### `--port <port>`

Specify the port to use. Default is `5173`.

<Terminal>
pyra dev --port 3000
</Terminal>

If the port is already in use, Pyra will automatically try the next available port.

### `--open [path]`

Automatically open the app in your browser on server start.

<Terminal>
pyra dev --open
</Terminal>

Open a specific path:

<Terminal>
pyra dev --open /docs
</Terminal>

### `--cors`

Enable CORS for all origins. Useful when your frontend and backend are on different ports.

<Terminal>
pyra dev --cors
</Terminal>

### `--strictPort`

Exit if the specified port is already in use instead of trying the next available port.

<Terminal>
pyra dev --port 5173 --strictPort
</Terminal>

### `--force`

Force Pyra to re-optimize dependencies, ignoring the cache.

<Terminal>
pyra dev --force
</Terminal>

<Callout type="note">
Use `--force` when you've added new dependencies or are experiencing caching issues.
</Callout>

### `--clearScreen`

Allow/disable clearing the terminal screen when logging. Default is `true`.

<Terminal>
pyra dev --clearScreen false
</Terminal>

## How It Works

When you run `pyra dev`, Pyra:

<Step number={1}>
**Starts the dev server** — Typically in under 100ms
</Step>

<Step number={2}>
**Pre-bundles dependencies** — Converts CommonJS modules to ESM and caches them
</Step>

<Step number={3}>
**Serves source files** — Your code is served as native ES modules, no bundling needed
</Step>

<Step number={4}>
**Watches for changes** — File changes trigger instant HMR updates
</Step>

### Dependency Pre-Bundling

Pyra pre-bundles dependencies using esbuild for two reasons:

1. **Convert CommonJS to ESM** — Many npm packages ship CommonJS, but browsers need ESM
2. **Reduce HTTP requests** — Packages with many internal modules (like lodash-es) are bundled into a single file

<GradientCode language="typescript">
// Before pre-bundling (100+ requests)
import { debounce } from 'lodash-es';

// After pre-bundling (1 request)
import { debounce } from '/.pyra/deps/lodash-es.js';
</GradientCode>

<Callout type="tip">
Pre-bundled dependencies are cached in `node_modules/.pyra`. Delete this folder to force a rebuild.
</Callout>

### Hot Module Replacement

Pyra's HMR is built on native ES modules and provides:

- **Fast updates** — Only changed modules are reloaded
- **State preservation** — React/Vue state is maintained across updates
- **Precise invalidation** — Only affected modules are updated

<GradientCode language="typescript">
// HMR API (usually handled by framework plugins)
if (import.meta.hot) {
  import.meta.hot.accept((newModule) => {
    // Handle module update
  });
}
</GradientCode>

## Configuration

You can configure the dev server in `pyra.config.ts`:

<GradientCode language="typescript">
import { defineConfig } from 'pyra';

export default defineConfig({
  server: {
    port: 3000,
    host: true,
    open: true,
    cors: true,
    strictPort: false,
    hmr: {
      overlay: true,
      port: 24678,
    },
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
    },
  },
});
</GradientCode>

See **[Server Options](/docs/configuration/server-options)** for full configuration details.

## Keyboard Shortcuts

While the dev server is running, press:

- **`r`** — Restart the server
- **`u`** — Show server URLs
- **`o`** — Open in browser
- **`c`** — Clear console
- **`q`** — Quit

## Troubleshooting

### Port Already in Use

If you see `Port 5173 is already in use`, either:

1. Stop the other process using that port
2. Use a different port: `pyra dev --port 3000`
3. Let Pyra auto-increment: Pyra will try 5174, 5175, etc.

### Slow Initial Load

If the first page load is slow:

1. Check your network tab — are there many requests?
2. Run `pyra dev --force` to rebuild the dependency cache
3. Consider adding slow dependencies to `optimizeDeps.include` in your config

### HMR Not Working

If HMR stops working:

1. Check the browser console for errors
2. Restart the dev server
3. Clear `node_modules/.pyra` and restart

<Callout type="warn">
Some code changes (like modifying `pyra.config.ts`) require a full server restart. Press `r` or restart manually.
</Callout>
